# Deploying

This app is **host-agnostic**. The game data is baked into the build
(`src/lib/baked-data.json`, regenerated by `bun run sync`), so there is **no
runtime database** — it deploys to any static/serverless/Node host with no DB to
provision. Most pages are static HTML; only `/hideout` and `/b/[code]` render on
demand.

The repo ships a **Cloudflare Workers** config as one documented option (below).
You are not tied to it — pick whatever host you like.

## Updating the data

Game data only changes when Valve patches. To refresh it:

```bash
bun run sync   # pulls deadlock-api, rewrites src/lib/baked-data.json
git commit -am "data: sync"   # commit the regenerated data
```

Redeploy (or let your host auto-deploy on push). A scheduled GitHub Action can
do this automatically — see `.github/workflows/`.

---

## Option A — Cloudflare Workers (the config in this repo)

Uses the [OpenNext](https://opennext.js.org/cloudflare) adapter
(`@opennextjs/cloudflare`). Config lives in `wrangler.jsonc` + `open-next.config.ts`.

**One-time setup**
1. A Cloudflare account (free tier is plenty — static assets and bandwidth are
   free; only Worker requests are metered, and you'd need tens of thousands of
   daily visitors to leave the free plan).
2. `bunx wrangler login`.

**Deploy**
```bash
bun run cf:preview   # build + run in the Cloudflare runtime locally (localhost)
bun run cf:deploy    # build + publish to Cloudflare
```

Or connect the GitHub repo in the Cloudflare dashboard (**Workers › Builds**) for
push-to-deploy with per-branch preview URLs.

**Custom domain:** in the Worker's settings add `fairfax.industries` (or your
domain). If Cloudflare already serves your DNS, it's a couple of clicks.

Requirements baked into `wrangler.jsonc`: the `nodejs_compat` flag and a recent
compatibility date. Needs **Next.js ≥ 16.2.6** (the adapter's floor).

## Option B — any Node or Bun server (Fly.io / Railway / Render / a VPS)

No adapter needed — it's a standard Next.js app:

```bash
bun run build
bun run start   # serves on $PORT
```

Deploy the repo as-is; no persistent disk or database required (the data is in
the bundle). Update data by redeploying after a `bun run sync` commit.

## Option C — Vercel

Import the GitHub repo; it builds and deploys with zero extra config (no runtime
DB to wire up). Data updates the same way — push a re-synced `baked-data.json`.

---

*Local development is unchanged: `bun run dev`. The SQLite database (`deadlock.db`)
and `bun:sqlite` are used only by `bun run sync` and the test suite — never at
runtime.*
